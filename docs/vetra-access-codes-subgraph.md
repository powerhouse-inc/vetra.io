# Access codes as a Switchboard subgraph

Report on moving the early-access / invite-code system off vetra.to's
self-hosted Postgres and into a GraphQL **subgraph** on the Switchboard
instance that already backs vetra.to (the `@powerhousedao/vetra-cloud-package`
reactor). The subgraph owns its own relational tables and gates the
management mutations behind Switchboard's existing auth (admin allowlist),
so vetra.to keeps no database of its own.

---

## 1. What the branch does today (`feat-add-early-access-gate`)

The branch implements the gate end-to-end against a Postgres database that
vetra.to connects to directly:

- **Schema** — `migrations/0001_invite_codes.mts` creates two tables:
  - `invite_codes` (`code` PK, `label`, `active`, `expires_at`,
    `max_uses`, `created_at`).
  - `invite_redemptions` (`code` + `user_did` composite PK,
    `redeemed_at`, `access_expires`; index on `user_did`).
- **DB access** — `modules/invites/lib/db.ts` builds a singleton Kysely
  instance over a `pg` `Pool` from `DATABASE_URL`. Schema types are
  generated by `kysely-codegen` into `modules/invites/lib/schema.ts`.
- **Logic** — `modules/invites/lib/codes.ts`:
  - `isCodeUsable(code)` — exists, `active`, not expired.
  - `redeemCode(code, did)` — transaction with `FOR UPDATE`; upserts a
    redemption with `access_expires = now() + 30 days`; idempotent.
  - `getAccessStatus(did)` — most-recent non-expired redemption + cohort.
- **Identity** — `modules/invites/lib/verify-identity.ts` verifies a
  Renown bearer token with `verifyAuthBearerToken()` from `@renown/sdk`
  and derives a canonical `did:pkh:<networkId>:<chainId>:<address>`. The
  client never asserts its own DID.
- **API routes** — `app/api/invite/{validate,redeem,status}/route.ts`,
  each `force-dynamic`, request bodies validated with Zod.
- **Admin tooling** — `scripts/invite-codes.mts`
  (`pnpm invite-codes add|list|redemptions|disable|enable`) talks to the
  DB directly. `scripts/migrate.mts` runs migrations.
- **Frontend** — `app/studio/early-access-gate.tsx`, a
  `gate → login → granted` state machine that calls the three routes and
  bridges the Renown login redirect via `sessionStorage`.
- **Env** — `DATABASE_URL`, `NEXT_PUBLIC_RENOWN_URL`,
  `NEXT_PUBLIC_STUDIO_URL` added to `.env.example` and the env schema.

The piece this report replaces is the **storage + server logic** (tables,
Kysely, the three API routes, and the direct-DB CLI). The Renown
verification model and the frontend state machine carry over almost
unchanged — only the endpoints they call move.

## 2. Why a subgraph, and why this reactor

vetra.to already points at a Switchboard instance whose runtime is
`@powerhousedao/vetra-cloud-package`. That package **already ships two
relational-DB-backed subgraphs** built on the exact pattern we need —
`vetra-cloud-secrets` and `vetra-cloud-observability` — so a third
`vetra-access-codes` subgraph is incremental, not new ground.

Benefits of folding invites into that reactor:

- **No second database for vetra.to.** The subgraph reuses the reactor's
  Postgres via `this.relationalDb.createNamespace("vetra-access-codes")`,
  which gives the subgraph its own isolated table namespace. vetra.to
  drops `pg`, `kysely`, `DATABASE_URL`, the migration runner, and the
  direct-DB CLI.
- **Auth is already solved at the gateway.** Switchboard verifies the
  Renown bearer token once, per request, and hands resolvers
  `context.user.address` plus `context.isAdmin(address)`. The
  per-route token verification in the branch collapses into the
  gateway. (Details in §5.)
- **One GraphQL endpoint.** vetra.to's frontend already speaks GraphQL to
  Switchboard; invites become queries/mutations on the same endpoint
  instead of bespoke Next.js routes.

### Relation to the `relational-db-subgraph` recipe

The [recipe](https://github.com/powerhouse-inc/recipes/tree/main/relational-db-subgraph)
demonstrates the canonical relational-DB-on-a-reactor layout: a Kysely
`schema.ts`, `migrations.ts` (`up`/`down`), a typed `query.ts`, and a
GraphQL surface — backed by namespaced relational storage and composed
into the supergraph.

One difference matters for us. The recipe's example is **processor-driven**:
a `RelationalDbProcessor` watches documents flowing through the reactor and
maintains a *denormalized read model* of them. That shape assumes the source
of truth is a document model. We deliberately **don't** want a document
model for codes (see §6), so we follow the recipe's relational/Kysely/GraphQL
structure but make the subgraph the **owner** of its tables (it writes them
directly from mutations), with no processor and no backing document. The
in-repo `vetra-cloud-secrets` subgraph is exactly this owner-of-its-tables
shape and is the closest working precedent — it's the template below.

## 3. Subgraph implementation

Layout (mirrors `subgraphs/vetra-cloud-secrets/` in `vetra-cloud-package`):

```
vetra-cloud-package/subgraphs/vetra-access-codes/
  index.ts            # BaseSubgraph subclass; onSetup wires DB + resolvers
  schema.ts           # GraphQL SDL (typeDefs)
  resolvers.ts        # query/redeem (public) + admin mutations (gated)
  db/
    schema.ts         # Kysely table interfaces
    migrations.ts     # idempotent up()
    codes.ts          # the validate/redeem/status logic (ported from vetra.to)
```

### 3.1 Subgraph class — `index.ts`

```typescript
import { BaseSubgraph } from "@powerhousedao/reactor-api";
import type { DocumentNode } from "graphql";
import type { Kysely } from "kysely";
import { schema } from "./schema.js";
import { createResolvers } from "./resolvers.js";
import { up } from "./db/migrations.js";
import type { VetraAccessCodesDB } from "./db/schema.js";

export class VetraAccessCodesSubgraph extends BaseSubgraph {
  name = "vetra-access-codes";
  typeDefs: DocumentNode = schema;
  resolvers: Record<string, unknown> = {};
  additionalContextFields = {};

  async onSetup() {
    const db = (await this.relationalDb.createNamespace(
      "vetra-access-codes",
    )) as unknown as Kysely<VetraAccessCodesDB>;

    await up(db as Kysely<any>);
    this.resolvers = createResolvers(db);
  }
}
```

`createNamespace` returns a Kysely instance scoped to this subgraph's own
tables; `up()` runs idempotently on every boot (the secrets subgraph does
the same), so there's no separate migration step to operate.

### 3.2 DB schema — `db/schema.ts`

Same shape the branch already settled on:

```typescript
export interface InviteCodes {
  code: string;            // PK, normalized lowercase
  label: string | null;
  active: boolean;
  expires_at: string | null;
  max_uses: number | null;
  created_at: string;
}

export interface InviteRedemptions {
  code: string;            // FK -> invite_codes.code
  user_did: string;        // did:pkh:...
  redeemed_at: string;
  access_expires: string | null;
}

export interface VetraAccessCodesDB {
  invite_codes: InviteCodes;
  invite_redemptions: InviteRedemptions;
}
```

### 3.3 Migrations — `db/migrations.ts`

Port `migrations/0001_invite_codes.mts` to the `up(db)` Kysely form used by
the secrets subgraph (`.ifNotExists()`, composite PK, the `user_did`
index). No `down()` is required for the boot-time runner.

### 3.4 Logic — `db/codes.ts`

`isCodeUsable`, `redeemCode`, `getAccessStatus` move over **verbatim** from
`modules/invites/lib/codes.ts` — they're plain Kysely against the same
schema. The only change is they receive the namespaced `db` instead of
calling `getDb()`.

### 3.5 GraphQL surface — `schema.ts`

```graphql
type AccessStatus {
  allowed: Boolean!
  code: String
  label: String
  accessExpires: String
}

type InviteCode {
  code: String!
  label: String
  active: Boolean!
  expiresAt: String
  maxUses: Int
  createdAt: String!
  redemptions: Int!
}

type Query {
  # public: does this code currently work? (validate-only, never consumes)
  inviteCodeValid(code: String!): Boolean!
  # caller-scoped: my access status, derived from the authed DID
  myAccessStatus: AccessStatus!
  # admin only: list codes + redemption counts
  inviteCodes: [InviteCode!]!
}

type Mutation {
  # caller-scoped: redeem a code for the authenticated DID
  redeemInviteCode(code: String!): AccessStatus!
  # admin only:
  createInviteCode(code: String!, label: String, expiresAt: String, maxUses: Int): InviteCode!
  setInviteCodeActive(code: String!, active: Boolean!): InviteCode!
}
```

Note how identity is sourced: `myAccessStatus`, `redeemInviteCode` take **no
DID argument** — the DID comes from the verified token in `context.user`, so
a caller can only ever act as themselves. The branch derived the DID
server-side for the same reason; here the gateway does it.

### 3.6 Resolvers — `resolvers.ts` (the auth-gated part)

```typescript
import type { Kysely } from "kysely";
import type { VetraAccessCodesDB } from "./db/schema.js";
import { isCodeUsable, redeemCode, getAccessStatus } from "./db/codes.js";

// Shape injected by reactor-api into every resolver's context (see §5).
type AuthContext = {
  user?: { address: string; chainId: number; networkId: string };
  isAdmin?: (address: string) => boolean;
};

function callerDid(ctx: AuthContext): string | null {
  const u = ctx.user;
  if (!u) return null;
  return `did:pkh:${u.networkId}:${u.chainId}:${u.address.toLowerCase()}`;
}

function requireAdmin(ctx: AuthContext): void {
  const addr = ctx.user?.address?.toLowerCase();
  if (!addr || !(ctx.isAdmin?.(addr) ?? false)) throw new Error("FORBIDDEN");
}

export function createResolvers(db: Kysely<VetraAccessCodesDB>): Record<string, any> {
  return {
    Query: {
      inviteCodeValid: (_p, { code }) => isCodeUsable(db, code),

      myAccessStatus: (_p, _a, ctx: AuthContext) => {
        const did = callerDid(ctx);
        if (!did) throw new Error("UNAUTHENTICATED");
        return getAccessStatus(db, did);
      },

      inviteCodes: async (_p, _a, ctx: AuthContext) => {
        requireAdmin(ctx);
        // join codes -> redemption counts (port the CLI `list` query)
        return listCodesWithCounts(db);
      },
    },

    Mutation: {
      redeemInviteCode: async (_p, { code }, ctx: AuthContext) => {
        const did = callerDid(ctx);
        if (!did) throw new Error("UNAUTHENTICATED");
        const result = await redeemCode(db, code, did);
        if (!result.ok) throw new Error("INVALID_CODE");
        return getAccessStatus(db, did);
      },

      createInviteCode: async (_p, args, ctx: AuthContext) => {
        requireAdmin(ctx);
        return insertCode(db, args); // normalize + onConflict doNothing
      },

      setInviteCodeActive: async (_p, { code, active }, ctx: AuthContext) => {
        requireAdmin(ctx);
        return setActive(db, code, active);
      },
    },
  };
}
```

This is the central point the goal calls out: **mutations that manage codes
(`createInviteCode`, `setInviteCodeActive`) and the admin listing are gated
with `requireAdmin`**, while `redeemInviteCode` / `myAccessStatus` are
gated to "the authenticated caller, acting only as themselves." This is the
same `ctx.user` / `ctx.isAdmin` pattern the `vetra-cloud-observability`
subgraph already uses for owner/admin-scoped queries and mutations.

### 3.7 Registration

Two files, matching how the existing subgraphs register:

- `subgraphs/index.ts` — add
  `export * as VetraAccessCodesSubgraph from "./vetra-access-codes/index.js";`
- `powerhouse.manifest.json` — add an entry under `subgraphs`:


## 4. What changes in vetra.to

The frontend keeps its `gate → login → granted` state machine; only the
calls change from Next.js routes to Switchboard GraphQL, with the Renown
bearer token sent as the standard `Authorization: Bearer <token>` header
(the gateway verifies it instead of each route doing so):

| Branch route (deleted)      | Replaced by GraphQL                          |
| --------------------------- | -------------------------------------------- |
| `POST /api/invite/validate` | `query inviteCodeValid(code)`                |
| `POST /api/invite/redeem`   | `mutation redeemInviteCode(code)` + Bearer   |
| `POST /api/invite/status`   | `query myAccessStatus` + Bearer              |

Removed from vetra.to: `modules/invites/lib/{db,schema,verify-identity}.ts`,
`migrations/`, `scripts/{migrate,invite-codes}.mts`, the `pg`/`kysely`/
`kysely-codegen` deps, the `migrate` / `codegen:invites` / `invite-codes`
scripts, and `DATABASE_URL`. `verify-identity.ts` is no longer needed in
vetra.to because the gateway performs the same `verifyAuthBearerToken`
check — though its logic effectively moves into Switchboard's auth service,
which already does exactly this.

Kept: the gate component, the `sessionStorage` redirect bridge, the
`code → login → redeem` ordering, the 30-day window, and Renown login.

A subtlety worth carrying over: today vetra.to can `validate` a code before
the user authenticates (the gate checks the code, *then* asks for login).
`inviteCodeValid` is therefore intentionally **public** (no auth) so the
same pre-login UX works; only redemption and admin actions require a token.

## 5. Auth model (how mutations are protected)

Switchboard's `@powerhousedao/reactor-api` builds the resolver context per
request and verifies the token at the gateway, so subgraph code never parses
headers:

- The gateway reads `Authorization: Bearer <token>` and verifies it with
  `verifyAuthBearerToken()` from `@renown/sdk` — the **same** call the
  branch uses in `verify-identity.ts`. From the verified credential it sets
  `context.user = { address, chainId, networkId }`.
- It injects `context.isAdmin(address)`, which returns
  `admins.includes(address.toLowerCase())` when auth is enabled.
- The admin allowlist comes from Switchboard env vars: **`ADMINS`** (and/or
  `ADMIN_USERS`), a comma-separated list of lowercased addresses. Auth
  itself is toggled by **`AUTH_ENABLED`**.

Operational caveat — **when `AUTH_ENABLED` is not `true`, `isAdmin()`
returns `true` for everyone and `context.user` is `undefined`.** So the
vetra-access-codes subgraph is only actually protected on an instance running
with `AUTH_ENABLED=true` and a populated `ADMINS` list. That must be set on
the vetra.io-backing Switchboard deployment for the management mutations to
be safe. (`SKIP_CREDENTIAL_VERIFICATION` likewise weakens verification and
must stay off in production.)

Granting an admin = adding their wallet address to `ADMINS` on the
Switchboard deployment. No code change, no redeploy of the subgraph.

## 6. Admin workflow for managing codes

With the subgraph in place, admin management is **GraphQL mutations on the
Switchboard endpoint, authenticated with an admin's Renown token** — no
direct DB access and no shell on the server.

A code can be created/disabled by any of:

1. **GraphQL client / curl** with an admin bearer token:
   ```bash
   curl -s "$SWITCHBOARD_URL/graphql" \
     -H "Authorization: Bearer $ADMIN_RENOWN_TOKEN" \
     -H 'content-type: application/json' \
     -d '{"query":"mutation{ createInviteCode(code:\"local-first\", label:\"Local-First Conf\", maxUses:200){ code active } }"}'
   ```
2. **A thin re-pointed CLI.** The branch's `scripts/invite-codes.mts` can be
   kept as an admin convenience, but rewritten to call the GraphQL mutations
   (with a token) instead of opening a `pg` pool. This preserves the
   `add | list | disable | enable` ergonomics while removing vetra.to's DB
   dependency.
3. **A future admin UI** (see §7) issuing the same mutations.

Listing and auditing:

- `query inviteCodes` (admin-gated) returns codes with redemption counts —
  the equivalent of the CLI's `list`.
- `query redemptions(code, address)` (admin-gated) returns which wallet (DID)
  redeemed which code and when — the equivalent of the CLI's `redemptions`
  view. Filter by `code`, by wallet `address` (to see whether a given address
  has ever used a code, on any chain), or omit both for the full audit.

Revoking access:

- `mutation revokeAccess(address)` (admin-gated) expires a wallet's
  currently-valid redemptions (sets `access_expires` to now) and returns how
  many grants were revoked. The audit rows are kept. Note this is not a hard
  ban: re-redeeming the *same* code is a no-op (stays revoked), but a
  *different* still-active code would grant fresh access — disable the
  relevant codes (`setInviteCodeActive(code, false)`) for a full lockout.
- `setInviteCodeActive(code, false)` only blocks *future* redemptions of that
  code; it does not touch access already granted to addresses that redeemed
  it. Use `revokeAccess` to remove an address's current access.

Prerequisite: the admin's wallet address is in the Switchboard `ADMINS`
list and they hold a valid Renown session to mint the bearer token.

## 7. Suggested future improvement: a document model

Per the goal, **no document model is required** — the subgraph owning plain
relational tables is the lean path and matches the secrets subgraph. As a
later step, modeling codes as a Powerhouse **document model** would add:

- **Auditability / history.** Document models are event-sourced, so every
  create/disable/redeem becomes a replayable operation with provenance,
  rather than a mutable row. The relational tables would then be a
  *processor-derived read model* (the recipe's original shape) instead of
  the source of truth.
- **Reuse of document-level permissions.** Switchboard exposes a
  `documentPermissionService`; codes-as-documents could lean on that
  instead of the env-var admin allowlist.
- **Editors / Connect UI** for managing codes visually.

The honest tradeoff: documents today have **weak access control and no
field obfuscation** — anyone who can read the drive can read the document's
contents. That's precisely why the **subgraph-owned table + auth-gated
mutations** approach is the right call now: the subgraph is the access
boundary. The data lives in tables the subgraph controls, the GraphQL
surface only exposes what resolvers choose to, and the admin mutations sit
behind `requireAdmin`. Codes never become readable just because someone can
read a drive. If/when document auth and field-level obfuscation mature, the
underlying storage can migrate to a document model behind the same GraphQL
surface without the frontend noticing.

## 8. Migration checklist

1. Scaffold `subgraphs/vetra-access-codes/` in `vetra-cloud-package` from
   the `vetra-cloud-secrets` template (§3).
2. Port schema + migrations + `codes.ts` logic from the branch.
3. Add the SDL and the gated resolvers; register in `subgraphs/index.ts`
   and `powerhouse.manifest.json`.
4. Set `AUTH_ENABLED=true` and `ADMINS=<addresses>` on the vetra.io-backing
   Switchboard deployment.
5. Re-point vetra.to's gate at the three GraphQL operations; delete the
   Next.js routes, the `pg`/Kysely layer, migrations, and `DATABASE_URL`.
6. Either retire `scripts/invite-codes.mts` or rewrite it to call the
   admin mutations with a token.
7. (Later) consider the document-model migration in §7.
